# Lab 11 – ConfigMaps, Secrets & Jobs, CronJobs

## 📖 Overview

This lab demonstrates how to externalize application configuration using **ConfigMaps**, securely manage sensitive information using **Secrets**, and automate recurring tasks with **Jobs** and **CronJobs**. You will also explore advanced integrations such as AWS Secrets Manager and Sealed Secrets for secure secret management in Kubernetes.

By completing this lab, you will learn how to:

* Create and use ConfigMaps.
* Store sensitive information in Kubernetes Secrets.
* Mount ConfigMaps as environment variables and files.
* Create and manage CronJobs.
* Update ConfigMaps dynamically.
* Decode Kubernetes Secrets.
* Integrate AWS Secrets Manager with Kubernetes.
* Automate backups to Amazon S3.
* Encrypt Kubernetes Secrets using Sealed Secrets.

# TASK SET 1 – Guided

## Exercise 1 – Create a ConfigMap

Create a ConfigMap containing application configuration values.

```bash
kubectl create configmap app-config \
--from-literal=DB_HOST=mysql \
--from-literal=APP_ENV=production \
-n dev
```

Verify the environment variable inside the running pod.

```bash
kubectl exec -it nginx-pod -n dev -- env | grep DB
```

> **📌 Note:** ConfigMaps store non-sensitive configuration data and can be mounted as environment variables or files.

---

## Exercise 2 – Create a Kubernetes Secret

Create a Secret containing the database password.

```bash
kubectl create secret generic db-secret \
--from-literal=DB_PASSWORD=SuperSecret123 \
-n dev
```

Update the pod specification to reference the Secret.

```yaml
env:
  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:
        name: db-secret
        key: DB_PASSWORD
```

> **💡 Tip:** Reference Secrets using `secretKeyRef` instead of hardcoding credentials inside application manifests.

---

## Exercise 3 – Create a CronJob

Create a CronJob that runs every minute.

After approximately three minutes, list the completed jobs.

```bash
kubectl get jobs -n dev
```

Verify that multiple completed Job instances have been created.

# TASK SET 2 – Practice

## Exercise 1 – Mount a ConfigMap as a File

Create a ConfigMap from an existing Nginx configuration file.

```bash
kubectl create configmap nginx-conf --from-file=nginx.conf -n dev
```

Update the pod specification.

```yaml
volumes:
  - name: config-vol
    configMap:
      name: nginx-conf

volumeMounts:
  - name: config-vol
    mountPath: /etc/nginx/custom.conf
    subPath: nginx.conf
```

Verify that the configuration file is available inside the container.

> **📌 Note:** Mounting a ConfigMap as a volume makes configuration files available without rebuilding the container image.

---

## Exercise 2 – Update a ConfigMap

Edit the ConfigMap.

```bash
kubectl edit configmap app-config -n dev
```

Observe the update behavior.

* ConfigMap volumes are automatically updated (typically within 60 seconds).
* Environment variables require the pod to be restarted before changes become available.

---

## Exercise 3 – Decode a Secret

Retrieve and decode the Secret value.

```bash
kubectl get secret db-secret -n dev \
-o jsonpath='{.data.DB_PASSWORD}' | base64 -d
```

> **⚠️ Warning:** Kubernetes Secrets are Base64-encoded, not encrypted. Consider additional encryption mechanisms for production environments.

# TASK SET 3 – Challenge

## Exercise 1 – Integrate AWS Secrets Manager

Add the External Secrets Helm repository.

```bash
helm repo add external-secrets https://charts.external-secrets.io
```

Install the External Secrets Operator.

```bash
helm install external-secrets \
external-secrets/external-secrets \
-n external-secrets \
--create-namespace
```

Create a **SecretStore** and **ExternalSecret** resource that retrieves secrets from AWS Secrets Manager.

> **💡 Tip:** External Secrets Operator synchronizes secrets from AWS Secrets Manager into native Kubernetes Secrets.

---

## Exercise 2 – Automate PVC Backups to Amazon S3

Create a CronJob that runs every day at **2:00 AM**.

Cron schedule:

```text
0 2 * * *
```

Example backup command.

```bash
aws s3 sync /data s3://my-backup-bucket/$(date +%F)/
```

> **📌 Note:** Ensure the workload has the required AWS IAM permissions to access the S3 bucket.

---

## Exercise 3 – Encrypt Secrets Using Sealed Secrets

Install Sealed Secrets.

```bash
helm install sealed-secrets sealed-secrets/sealed-secrets -n kube-system
```

Encrypt a Kubernetes Secret.

```bash
kubeseal --format yaml < secret.yaml > sealed-secret.yaml
```

Commit the encrypted Secret to Git.

```bash
git add sealed-secret.yaml

git commit -m "add sealed secret"
```

> **💡 Tip:** Sealed Secrets allow encrypted Kubernetes Secrets to be stored safely in version control systems.

# 💡 Best Practices

* Store non-sensitive configuration in ConfigMaps.
* Store sensitive information only in Kubernetes Secrets.
* Avoid hardcoding credentials in manifests or container images.
* Use External Secrets Operator for centralized secret management.
* Schedule automated backups for persistent data.
* Use Sealed Secrets when storing Kubernetes manifests in Git repositories.
* Apply the principle of least privilege to AWS IAM permissions.
* Rotate secrets regularly.

---

# 🔒 Security Considerations

* Never store plaintext credentials in Git repositories.
* Enable encryption at rest for Kubernetes Secrets where supported.
* Restrict access to Secrets using Kubernetes RBAC.
* Use AWS IAM Roles for Service Accounts (IRSA) when accessing AWS services.
* Regularly audit Secret usage and permissions.
* Secure backup locations such as Amazon S3 using encryption and access policies.
