# TASK SET 1 — Guided

## Exercise 1 — Create a Flask Application, `requirements.txt`, and Dockerfile

Create the following application files.

### 📄 app.py

```python
from flask import Flask

app = Flask(__name__)

@app.route('/')
def hello():
    return 'Hello Docker!'
```

---

### 📄 requirements.txt

```text
flask
```

---

### 📄 Dockerfile

```dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .

RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 5000

CMD ["python","-m","flask","run","--host=0.0.0.0"]
```

---

### 🔨 Build the Docker Image

```bash
docker build -t flask-app:1.0 .
```

---

### ▶️ Run the Docker Container

```bash
docker run -d -p 5000:5000 flask-app:1.0
```

---

## Exercise 2 — Test the Running Container

Verify that the application is running correctly.

```bash
curl localhost:5000
```

Expected output:

```text
Hello Docker!
```

---

## Exercise 3 — Tag and Push the Image to Docker Hub

### Tag the Docker Image

```bash
docker tag flask-app:1.0 <dockerhub-username>/flask-app:1.0
```

### Login to Docker Hub

```bash
docker login
```

### Push the Docker Image

```bash
docker push <dockerhub-username>/flask-app:1.0
```

> **📝 Note:** Replace `<dockerhub-username>` with your Docker Hub username.

# TASK SET 2 — Practice

## Exercise 1 — Observe Docker Layer Caching

Build the Docker image multiple times to observe Docker cache behavior.

### First Build

```bash
docker build -t myapp:1.0 .
```

### Build Without Cache

```bash
docker build -t myapp:1.0 --no-cache .
```

> ⚠️ This build will be slower because Docker rebuilds every layer.

### Build Again Using Cache

```bash
docker build -t myapp:1.0 .
```

> 💡 This build should be significantly faster because Docker reuses cached layers.

## Exercise 3 — Create a `.dockerignore` File

Reduce the Docker build context by excluding unnecessary files.

Create the `.dockerignore` file.

```bash

```

Rebuild the image.

```bash

```

> 💡 Compare the Docker build context size before and after adding `.dockerignore`.

# TASK SET 3 — Challenge

## Exercise 1 — Create a Multi-Stage Docker Build

Create a file named:

```text
Dockerfile.multistage
```

### Dockerfile.multistage

```dockerfile
FROM node:18 AS builder

WORKDIR /app

COPY package*.json .

RUN npm install

COPY . .

RUN npm run build

FROM nginx:alpine

COPY --from=builder /app/dist /usr/share/nginx/html
```

---

### Build the Multi-Stage Image

```bash
docker build -f Dockerfile.multistage -t app:prod .
```

Compare the resulting image sizes.

```bash
docker images
```

> 📌 Multi-stage builds help reduce the final image size by excluding unnecessary build dependencies.

## Exercise 2 — Scan Docker Images for CVEs

Use Docker Scout or Trivy to identify known vulnerabilities.

### Option 1 — Docker Scout

```bash
docker scout cves flask-app:1.0
```

### Option 2 — Trivy

```bash
trivy image flask-app:1.0
```

> 🔒 Regular vulnerability scanning helps identify and remediate security issues before deployment.

---

## Exercise 3 — Push the Image to Amazon ECR

### Create an Amazon ECR Repository

```bash
aws ecr create-repository --repository-name flask-app --region ap-south-1
```

---

### Authenticate Docker with Amazon ECR

```bash
aws ecr get-login-password \
| docker login --username AWS \
--password-stdin <account-id>.dkr.ecr.ap-south-1.amazonaws.com
```

---

### Tag the Docker Image

```bash
docker tag flask-app:1.0 <account-id>.dkr.ecr.ap-south-1.amazonaws.com/flask-app:1.0
```

---

### Push the Docker Image

```bash
docker push <account-id>.dkr.ecr.ap-south-1.amazonaws.com/flask-app:1.0
```

> **📝 Note:** Replace `<account-id>` with your AWS account ID.

# 💡 Best Practices

* Use lightweight base images (for example, `python:3.11-slim`) whenever possible.
* Minimize Docker image layers.
* Use `.dockerignore` to exclude unnecessary files.
* Leverage Docker layer caching to speed up builds.
* Implement multi-stage builds for production images.
* Scan container images regularly for vulnerabilities.
* Use non-root users in production containers.
* Add Docker health checks for production workloads.
* Store secrets securely; never embed credentials in Docker images.

---

# 🔒 Security Considerations

* Avoid running containers as the root user.
* Keep base images up to date.
* Remove unnecessary packages and tools from production images.
* Regularly scan images using Docker Scout or Trivy.
* Push images only to trusted registries such as Docker Hub or Amazon ECR.
