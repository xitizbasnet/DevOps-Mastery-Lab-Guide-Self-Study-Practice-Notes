# Lab 07 – Kubernetes Setup (Minikube or Amazon EKS)

## 📖 Overview

This lab introduces the fundamentals of setting up a Kubernetes cluster using either **Minikube** for local development or **Amazon Elastic Kubernetes Service (Amazon EKS)** for a managed cloud environment. You will verify cluster health, explore Kubernetes architecture, inspect nodes and system components, create namespaces, examine Kubernetes resources, and use debugging tools to understand cluster networking and resource utilization.

By completing this lab, you will learn how to:

* Start a local Kubernetes cluster with Minikube.
* Connect to an existing Amazon EKS cluster.
* Verify cluster health and node status.
* Explore Kubernetes system components.
* Inspect node configuration and capacity.
* Create and manage namespaces.
* Discover Kubernetes resource types.
* Read Kubernetes API documentation using `kubectl explain`.
* Deploy temporary debugging pods.
* Monitor cluster resource usage.
* Inspect Kubernetes control plane components.

---

## 🎯 Learning Objectives

After completing this lab, you should be able to:

* Deploy or connect to a Kubernetes cluster.
* Verify node readiness and cluster status.

# TASK SET 1 — Guided

## Exercise 1 – Start Minikube or Connect to Amazon EKS

### Option 1 – Start a Local Minikube Cluster

Start Minikube using Docker as the driver.

```bash
minikube start --driver=docker --cpus=2 --memory=4096
```

Verify that the cluster node is ready.

```bash
kubectl get nodes
```

---

### Option 2 – Connect to an Amazon EKS Cluster

Update the Kubernetes configuration file to connect to the EKS cluster.

```bash
aws eks update-kubeconfig --name devops-cluster --region ap-south-1
```

Verify the connected cluster.

```bash
kubectl get nodes
```

> **💡 Tip:** A node with the **Ready** status indicates that it is healthy and capable of scheduling workloads.

---

## Exercise 2 – List System Pods

Display all Kubernetes system pods running in the **kube-system** namespace.

```bash
kubectl get pods -n kube-system
```

Review the output and identify core Kubernetes components such as:

* CoreDNS
* kube-proxy
* etcd (control plane)
* kube-apiserver
* kube-controller-manager
* kube-scheduler

> **📌 Note:** The exact components displayed may vary depending on whether you are using Minikube or Amazon EKS.

---

## Exercise 3 – Inspect a Kubernetes Node

Display detailed information about a cluster node.

```bash
kubectl describe node
```

Review the following sections:

* Capacity
* Allocatable
* Conditions
* Addresses
* System Information
* Allocated Resources

> **💡 Tip:** Compare the **Capacity** and **Allocatable** values to understand the resources available for scheduling workloads.

# TASK SET 2 – Practice

## Exercise 1 – Create a Namespace

Create a namespace named **dev**.

```bash
kubectl create ns dev
```

List all namespaces.

```bash
kubectl get ns
```

Verify that the **dev** namespace appears in the output.

# TASK SET 3 – Challenge

## Exercise 1 – Deploy a Temporary Debug Pod

Create a temporary BusyBox container for troubleshooting.

```bash
kubectl run debug --image=busybox -it --rm -- sh
```

Inside the container, execute the following commands.

Display network interfaces.

```bash
ip addr
```

Resolve the Kubernetes service.

```bash
nslookup kubernetes
```

Test connectivity to an Nginx service.

```bash
wget -qO- http://nginx-svc
```

> **📌 Note:** The command succeeds only if the **nginx-svc** service exists within the cluster.

---

## Exercise 2 – Monitor Resource Usage

Install the Kubernetes Metrics Server.

```bash
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
```

Display resource usage for all nodes.

```bash
kubectl top nodes
```

Display resource usage for all pods across all namespaces.

```bash
kubectl top pods -A
```

> **💡 Tip:** Metrics Server must be running successfully before the `kubectl top` commands return resource statistics.

---

## Exercise 3 – Inspect the Kubernetes Scheduler

Describe the Kubernetes Scheduler pod.

```bash
kubectl describe pod -n kube-system $(kubectl get pods -n kube-system | grep scheduler | awk '{print $1}')
```

Review the following information:

* Scheduler image
* Startup command
* Startup flags
* Environment variables
* Mounted volumes
* Events

> **📌 Note:** Scheduler configuration may differ between Minikube and Amazon EKS because managed Kubernetes services abstract portions of the control plane.

# 💡 Best Practices

* Verify cluster health before deploying workloads.
* Organize applications using namespaces.
* Use `kubectl explain` to understand Kubernetes resources.
* Monitor cluster resource usage regularly.
* Remove temporary debugging pods after troubleshooting.
* Keep Kubernetes components and CLI tools updated.
* Limit access to cluster administration using role-based access control (RBAC).
* Regularly review node resource allocation to prevent resource exhaustion.

---

# 🔒 Security Considerations

* Follow the principle of least privilege when granting Kubernetes permissions.
* Restrict administrative access to trusted users.
* Secure Kubernetes credentials and kubeconfig files.
* Monitor cluster activity for unauthorized changes.
* Keep Minikube, Kubernetes, and Amazon EKS clusters updated with the latest security patches.
* Avoid exposing internal services unless required.
