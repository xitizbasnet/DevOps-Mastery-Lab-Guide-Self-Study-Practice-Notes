# Lab 08 – Pods, ReplicaSets & Deployments

## 📖 Overview

This lab introduces Kubernetes workload management using **Pods**, **ReplicaSets**, and **Deployments**. You will deploy an Nginx application, scale replicas, perform rolling updates, roll back deployments, configure Horizontal Pod Autoscaling (HPA), and implement health probes to improve application availability and reliability.

By completing this lab, you will learn how to:

* Deploy applications using Kubernetes Deployments.
* Manage ReplicaSets through Deployments.
* Scale applications horizontally.
* Perform rolling updates with zero downtime.
* Roll back failed deployments.
* Deploy multiple applications within the same namespace.
* Configure Horizontal Pod Autoscaling (HPA).
* Troubleshoot deployment failures.
* Configure liveness and readiness probes.

# TASK SET 1 – Guided

## Exercise 1 – Create an Nginx Deployment

Create a file named **nginx-deployment.yaml**.

```yaml id="ojjndu"
apiVersion: apps/v1
kind: Deployment

metadata:
  name: nginx-app
  namespace: dev

spec:
  replicas: 3

  selector:
    matchLabels:
      app: nginx

  strategy:
    type: RollingUpdate

    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0

  template:
    metadata:
      labels:
        app: nginx

    spec:
      containers:
      - name: nginx
        image: nginx:1.25

        ports:
        - containerPort: 80

        resources:
          requests:
            cpu: 100m
            memory: 64Mi

          limits:
            memory: 128Mi
```

Apply the deployment.

```bash id="afyv8x"
kubectl apply -f nginx-deployment.yaml
```

Verify that three pods are running.

```bash id="2e1jdh"
kubectl get pods -n dev
```

> **📌 Note:** The deployment automatically creates a ReplicaSet that maintains the desired number of pod replicas.

---

## Exercise 2 – Scale the Deployment

Scale the deployment from three replicas to five replicas.

```bash id="efp8c3"
kubectl scale deployment nginx-app --replicas=5 -n dev
```

Monitor pod creation.

```bash id="lq62k2"
kubectl get pods -n dev -w
```

> **💡 Tip:** The `-w` option continuously watches for changes until you stop the command.

---

## Exercise 3 – Perform a Rolling Update

Update the container image to **nginx:1.26**.

```bash id="5m27h6"
kubectl set image deployment/nginx-app nginx=nginx:1.26 -n dev
```

Monitor the rollout status.

```bash id="sg8oqv"
kubectl rollout status deployment/nginx-app -n dev
```

> **📌 Note:** Kubernetes replaces pods gradually according to the configured RollingUpdate strategy.

# TASK SET 2 – Practice

## Exercise 1 – Verify Automatic Pod Recovery

Delete one pod manually.

```bash id="a97vyo"
kubectl delete pod -n dev <pod-name>
```

Monitor pod recreation.

```bash id="ymc0je"
kubectl get pods -n dev -w
```

> **💡 Tip:** Replace `<pod-name>` with the name of one of the running Nginx pods.

> **📌 Note:** The ReplicaSet automatically recreates the deleted pod to maintain the desired replica count.

---

## Exercise 2 – Roll Back the Deployment

Restore the previous deployment version.

```bash id="v6js74"
kubectl rollout undo deployment/nginx-app -n dev
```

View the deployment revision history.

```bash id="o62x4t"
kubectl rollout history deployment/nginx-app -n dev
```

## Exercise 3 – Deploy a Second Application

Deploy an Apache HTTP Server application in the **dev** namespace.

```bash id="g4ef2f"
kubectl create deployment httpd-app --image=httpd:2.4 -n dev
```

Verify that both deployments are running.

```bash id="vv5gsi"
kubectl get deployments -n dev
```

> **📌 Note:** Multiple deployments can coexist within the same namespace without interfering with one another.

# TASK SET 3 – Challenge

## Exercise 1 – Configure Horizontal Pod Autoscaling

Create an HPA for the **nginx-app** deployment.

```bash id="v9t2n8"
kubectl autoscale deployment nginx-app -n dev \
--min=2 --max=10 --cpu-percent=50
```

Verify the HPA configuration.

```bash id="p63bd6"
kubectl get hpa -n dev
```

> **💡 Tip:** Metrics Server must be installed and running before HPA can collect CPU metrics.

---

## Exercise 2 – Simulate an Image Pull Failure

Update the deployment with a non-existent image.

```bash id="pjn4c0"
kubectl set image deployment/nginx-app nginx=nginx:doesnotexist -n dev
```

Verify the pod status.

```bash id="5g4uzp"
kubectl get pods -n dev
```

Expected output:

```text id="jzywdq"
ImagePullBackOff
```

Restore the correct image.

```bash id="2qmfaj"
kubectl set image deployment/nginx-app nginx=nginx:1.25 -n dev
```

> **⚠️ Warning:** Incorrect image names or unavailable container registries commonly result in the **ImagePullBackOff** status.

---

## Exercise 3 – Configure Liveness and Readiness Probes

Update the deployment YAML by adding the following health probes under the container specification.

```yaml id="8q3blq"
livenessProbe:
  httpGet:
    path: /
    port: 80
  initialDelaySeconds: 10
  periodSeconds: 10

readinessProbe:
  httpGet:
    path: /
    port: 80
  initialDelaySeconds: 5
  periodSeconds: 5
```

Apply the updated deployment.

```bash id="awjlwm"
kubectl apply -f nginx-deployment.yaml
```

> **📌 Note:** Liveness probes determine whether a container should be restarted, while readiness probes determine whether a pod is ready to receive traffic.

# 💡 Best Practices

* Use Deployments instead of creating Pods directly.
* Configure resource requests and limits for every container.
* Use rolling updates to minimize application downtime.
* Enable HPA for applications with variable workloads.
* Configure liveness and readiness probes for production workloads.
* Monitor rollout status before proceeding with additional updates.
* Use meaningful labels and selectors for workload management.
* Keep deployment manifests under version control.

---

# 🔒 Security Considerations

* Use trusted container images from verified registries.
* Apply resource limits to prevent resource exhaustion.
* Restrict access to Kubernetes namespaces using RBAC.
* Regularly update application images with security patches.
* Monitor deployment events and pod status for failures.
* Scan container images for vulnerabilities before deployment.
