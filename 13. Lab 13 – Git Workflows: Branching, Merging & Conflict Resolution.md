# Lab 13 – Git Workflows: Branching, Merging & Conflict Resolution

This lab introduces professional Git workflows used in collaborative software development. * Resolve merge conflicts.

* Implement the Git Flow branching model.
* Configure GitHub branch protection.
* Enforce Conventional Commits using Git hooks.

# TASK SET 1 – Guided

## Exercise 1 – Clone a Repository and Create a Feature Branch

Clone the repository.

```bash
git clone https://github.com/<username>/devops-journey.git
```

Navigate to the repository.

```bash
cd devops-journey
```

Create and switch to a new feature branch.

```bash
git checkout -b feature/hello
```

Create a Python file.

```bash
echo 'print("Hello")' > hello.py
```

Stage and commit the changes.

```bash
git add .

git commit -m "feat: add hello.py"
```

Push the branch to GitHub.

```bash
git push origin feature/hello
```

> **📝 Note:** Replace `<username>` with the appropriate GitHub username or organization.

---

## Exercise 2 – Create a Pull Request

Open GitHub and create a Pull Request.

1. Navigate to the repository.
2. Select **New Pull Request**.
3. Compare **feature/hello** with **main**.
4. Add a meaningful title and description.
5. Create the Pull Request.
6. Merge the Pull Request after review.

> **💡 Tip:** Pull Requests provide an opportunity for code review before changes are merged into the main branch.

## Exercise 3 – Update the Local Repository

Switch to the main branch.

```bash
git checkout main
```

Pull the latest changes.

```bash
git pull origin main
```

Verify that the new file exists.

```bash
ls -la hello.py
```

# TASK SET 2 – Practice

## Exercise 1 – Resolve a Merge Conflict

### Developer 1

Create a new branch.

```bash
git checkout -b dev1
```

Modify the shared file.

```bash
echo "line A" > shared.txt
```

Commit the changes.

```bash
git add shared.txt

git commit -m "dev1 change"
```

---

### Developer 2

Create another branch from the main branch.

```bash
git checkout -b dev2 main
```

Modify the same file.

```bash
echo "line B" > shared.txt
```

Commit the changes.

```bash
git add shared.txt

git commit -m "dev2 change"
```

Merge the first branch.

```bash
git merge dev1
```

Resolve the conflict by editing the file.

```bash
nano shared.txt
```

Remove the conflict markers and keep the desired content.

Stage and commit the resolved file.

```bash
git add shared.txt

git commit -m "resolve: merge conflict"
```

> **⚠️ Warning:** Conflict markers (`<<<<<<<`, `=======`, and `>>>>>>>`) must be removed before committing the resolved file.

## Exercise 2 – Use Git Stash

Modify a file.

```bash
nano app.py
```

Temporarily save the uncommitted changes.

```bash
git stash
```

Switch to the main branch and update it.

```bash
git checkout main

git pull
```

Restore the stashed changes.

```bash
git stash pop
```

> **💡 Tip:** Git stash is useful when you need to switch tasks without committing incomplete work.

---

## Exercise 3 – Revert a Commit

View the commit history.

```bash
git log --oneline
```

Identify the commit SHA to revert.

Create a new commit that reverses the selected commit.

```bash
git revert <commit-sha>
```

> **📝 Note:** Replace `<commit-sha>` with the SHA of the commit you want to undo.

> **📌 Note:** `git revert` preserves repository history by creating a new commit that reverses the selected changes.

# TASK SET 3 – Challenge

## Exercise 1 – Implement Git Flow

Create the **develop** branch.

```bash
git checkout -b develop
```

Create a feature branch.

```bash
git checkout -b feature/login develop
```

After completing development, merge the feature into the develop branch.

```bash
git checkout develop

git merge feature/login
```

Create a release branch.

```bash
git checkout -b release/1.0 develop
```

Merge the release into the main branch.

```bash
git checkout main

git merge release/1.0
```

Create a version tag.

```bash
git tag v1.0
```

> **💡 Tip:** Git Flow separates ongoing development, releases, and production hotfixes into dedicated branches.

## Exercise 2 – Configure Branch Protection

Configure branch protection rules in GitHub.

1. Navigate to **Repository Settings**.
2. Select **Branches**.
3. Create a new branch protection rule.
4. Select the **main** branch.
5. Enable **Require a pull request before merging**.
6. Enable **Require pull request reviews before merging**.
7. Enable **Require status checks to pass before merging**.
8. Save the rule.

> **📌 Note:** Branch protection helps prevent direct changes to production branches.

## Exercise 3 – Create a Commit Message Hook

Create the **commit-msg** hook.

```bash
nano .git/hooks/commit-msg
```

Add the following script.

```bash
#!/bin/bash

MSG=$(cat $1)

if ! echo "$MSG" | grep -qE '^(feat|fix|chore|docs|style|refactor|test|ci)(\(.+\))?: .+'; then
    echo "ERROR: Commit must follow Conventional Commits format"
    exit 1
fi
```

Make the hook executable.

```bash
chmod +x .git/hooks/commit-msg
```

The hook validates commit messages before allowing commits.

> **💡 Tip:** Conventional Commits improve commit consistency and support automated release tooling.

# 💡 Best Practices

* Create feature branches for all new development.
* Keep commits small and focused.
* Write meaningful commit messages using the Conventional Commits standard.
* Review Pull Requests before merging.
* Rebase or merge regularly to minimize conflicts.
* Use Git stash for temporary work instead of creating unnecessary commits.
* Protect production branches using GitHub branch protection rules.
* Tag production releases for easier version tracking.

---

# 🔒 Security Considerations

* Protect the main branch from direct commits.
* Require Pull Request reviews before merging.
* Require successful CI checks before deployment.
* Use signed commits where appropriate.
* Restrict repository permissions based on user roles.
* Regularly audit repository access and branch protection settings.
