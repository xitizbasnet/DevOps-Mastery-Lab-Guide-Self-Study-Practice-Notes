# Lab 09 – Services & Load Balancing

## 📖 Overview

This lab introduces Kubernetes **Services** and load balancing mechanisms. You will expose applications using **ClusterIP**, **NodePort**, and **LoadBalancer** services, verify service discovery using Kubernetes DNS, and implement advanced networking concepts such as **Blue-Green Deployments**, **ExternalName Services**, and **Headless Services**.

By completing this lab, you will learn how to:

* Expose applications using NodePort services.
* Create internal ClusterIP services.
* Deploy external LoadBalancer services.
* Verify Kubernetes service discovery.
* Scale applications behind a service.
* Implement Blue-Green deployment strategies.
* Configure ExternalName services.
* Create and test Headless services.

# TASK SET 1 – Guided

## Exercise 1 – Create a NodePort Service

Create a file named **nginx-svc.yaml**.

```yaml
apiVersion: v1
kind: Service

metadata:
  name: nginx-svc
  namespace: dev

spec:
  type: NodePort

  selector:
    app: nginx

  ports:
    - port: 80
      targetPort: 80
      nodePort: 30080
```

Apply the service.

```bash
kubectl apply -f nginx-svc.yaml
```

Access the application.

```bash
curl http://<Node-IP>:30080
```

> **📝 Note:** Replace `<Node-IP>` with the IP address of your Minikube node or Kubernetes worker node.

---

## Exercise 2 – Inspect the Service

Display detailed information about the service.

```bash
kubectl describe svc nginx-svc -n dev
```

Review the following information:

* Service Type
* Cluster IP
* NodePort
* Selector
* Endpoints

> **💡 Tip:** The **Endpoints** section displays the IP addresses of all pods currently serving requests.

---

## Exercise 3 – Create a ClusterIP Service

Expose the Apache deployment internally.

```bash
kubectl expose deployment httpd-app --port=80 --type=ClusterIP -n dev
```

Create a temporary BusyBox pod.

```bash
kubectl run test --image=busybox -it --rm -- wget -qO- http://httpd-app
```

> **📌 Note:** ClusterIP services are only accessible from within the Kubernetes cluster.

# TASK SET 2 – Practice

## Exercise 1 – Convert the Service to LoadBalancer

Change the existing NodePort service into a LoadBalancer service.

```bash
kubectl patch svc nginx-svc -n dev -p '{"spec":{"type":"LoadBalancer"}}'
```

Monitor the service.

```bash
kubectl get svc nginx-svc -n dev -w
```

After an external IP is assigned, access the application.

```bash
curl http://<EXTERNAL-IP>
```

> **📝 Note:** Replace `<EXTERNAL-IP>` with the assigned external IP address. On Minikube, additional configuration such as `minikube tunnel` may be required.

---

## Exercise 2 – Test Kubernetes DNS

Launch a temporary BusyBox container.

```bash
kubectl run test -it --rm --image=busybox -- wget -qO- http://nginx-svc.dev.svc.cluster.local
```

Verify that the service is reachable using its fully qualified domain name (FQDN).

> **💡 Tip:** Kubernetes automatically creates DNS records for all services.

---

## Exercise 3 – Scale the Deployment

Increase the number of Nginx replicas.

```bash
kubectl scale deployment nginx-app --replicas=6 -n dev
```

Refresh the application repeatedly and observe traffic distribution across multiple pods.

> **📌 Note:** Kubernetes automatically load balances requests across all healthy pod endpoints associated with the service.

# TASK SET 3 – Challenge

## Exercise 1 – Implement a Blue-Green Deployment

Deploy version 1 of the application.

```bash
kubectl create deployment app-v1 --image=nginx:1.25 -n dev
```

Deploy version 2 of the application.

```bash
kubectl create deployment app-v2 --image=nginx:1.26 -n dev
```

Expose the Blue deployment.

```bash
kubectl expose deployment app-v1 --port=80 --name=app-blue -n dev
```

Expose the Green deployment.

```bash
kubectl expose deployment app-v2 --port=80 --name=app-green -n dev
```

Switch application traffic by updating the selector on the primary service to point to either the Blue or Green deployment.

> **💡 Tip:** Blue-Green deployments enable rapid switching between application versions with minimal downtime.

---

## Exercise 2 – Create an ExternalName Service

Create a file named **externalname-service.yaml**.

```yaml
apiVersion: v1
kind: Service

metadata:
  name: github-api
  namespace: dev

spec:
  type: ExternalName
  externalName: api.github.com
```

Apply the configuration.

```bash
kubectl apply -f externalname-service.yaml
```

> **📌 Note:** ExternalName services map Kubernetes service names to external DNS names without creating proxy endpoints.

---

## Exercise 3 – Create a Headless Service

Create a file named **headless-svc.yaml**.

```yaml
apiVersion: v1
kind: Service

metadata:
  name: nginx-headless
  namespace: dev

spec:
  clusterIP: None

  selector:
    app: nginx

  ports:
    - port: 80
      targetPort: 80
```

Apply the configuration.

```bash
kubectl apply -f headless-svc.yaml
```

Test DNS resolution.

```bash
kubectl run dns-test -it --rm --image=busybox -- nslookup nginx-headless.dev.svc.cluster.local
```

> **💡 Tip:** Unlike a standard service, a Headless Service returns the IP addresses of individual pods instead of a single virtual IP.

# 💡 Best Practices

* Use ClusterIP for internal-only communication.
* Use NodePort primarily for development and testing environments.
* Use LoadBalancer services in cloud environments.
* Prefer DNS-based service discovery instead of hardcoded IP addresses.
* Use Headless Services for StatefulSets and direct pod discovery.
* Implement Blue-Green deployments to minimize deployment risk.
* Label workloads consistently for reliable service selection.
* Monitor service endpoints to verify backend availability.

---

# 🔒 Security Considerations

* Expose only the services that require external access.
* Restrict access to NodePort and LoadBalancer services using firewall rules or security groups.
* Use Kubernetes Network Policies to limit communication between workloads.
* Protect externally exposed services using TLS and ingress controllers.
* Regularly review service configurations to prevent unintended exposure.
