# Lab 19 – Terraform Providers, Resources, State & Modules

## 📖 Overview

This lab introduces **Terraform**, an Infrastructure as Code (IaC) tool used to provision and manage AWS infrastructure. You will create AWS resources such as EC2 instances, VPCs, Security Groups, and S3 buckets, manage Terraform state locally and remotely, organize infrastructure using modules, work with Terraform workspaces, and validate deployments using Terratest.

By completing this lab, you will learn how to:

* Configure Terraform providers.
* Provision AWS infrastructure.
* Manage Terraform state.
* Store remote state in Amazon S3.
* Enable state locking with DynamoDB.
* Use Terraform modules.
* Manage multiple environments with workspaces.
* Test infrastructure deployments using Terratest.

# TASK SET 1 – Guided

## Exercise 1 – Create and Deploy an EC2 Instance

Create a file named **main.tf**.

```hcl id="j8r2dw"
terraform {
  required_version = ">= 1.5"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "ap-south-1"
}

resource "aws_instance" "web" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = "t2.micro"

  tags = {
    Name        = "tf-web"
    Environment = "dev"
  }
}

output "instance_ip" {
  value = aws_instance.web.public_ip
}
```

Initialize the Terraform project.

```bash id="e2wnxq"
terraform init
```

Validate the configuration.

```bash id="0aqztm"
terraform validate
```

Review the execution plan.

```bash id="v89k5p"
terraform plan
```

Deploy the infrastructure.

```bash id="b0rw76"
terraform apply
```

Record the EC2 instance ID after deployment.

## Exercise 2 – Configure Remote State

Create an Amazon S3 bucket and a DynamoDB table for remote state storage and state locking.

Update the Terraform backend configuration.

```hcl id="0o6f6k"
terraform {
  backend "s3" {
    bucket         = "my-tf-state-bucket"
    key            = "devops/terraform.tfstate"
    region         = "ap-south-1"
    dynamodb_table = "tf-state-lock"

  }
}
```

```bash id="mcbdbd"

```


## Exercise 3 – Validate Infrastructure Using Terratest

Create a file named **terraform_test.go**.

```go id="8j0npx"
package test

import (
	"testing"

	"github.com/gruntwork-io/terratest/modules/http-helper"
	"github.com/gruntwork-io/terratest/modules/terraform"
)

func TestEC2(t *testing.T) {

	opts := &terraform.Options{
		TerraformDir: "..",
	}

	defer terraform.Destroy(t, opts)

	terraform.InitAndApply(t, opts)

	ip := terraform.Output(t, opts, "instance_ip")

	http_helper.HttpGetWithRetry(
		t,
		"http://"+ip,
		nil,
		200,
		"nginx",
		10,
		5*time.Second,
	)
}
```

Run the Terratest.

Verify that:

* Terraform successfully provisions the infrastructure.
* The EC2 instance is running.
* The web server responds with HTTP status code **200**.

# 💡 Best Practices

* Store Terraform state remotely in Amazon S3.
* Enable DynamoDB state locking to prevent concurrent modifications.
* Organize infrastructure into reusable modules.
* Use variables instead of hard-coded values.
* Separate development and production environments using workspaces.
* Review execution plans before applying changes.
* Keep Terraform providers updated.
* Automate infrastructure validation using Terratest.

---

# 🔒 Security Considerations

* Encrypt remote Terraform state stored in Amazon S3.
* Restrict access to the Terraform state bucket using IAM policies.
* Enable versioning on the S3 state bucket.
* Protect sensitive Terraform variables using environment variables or secret managers.
* Apply the principle of least privilege for Terraform IAM users and roles.
* Never commit Terraform state files containing sensitive information to source control.
