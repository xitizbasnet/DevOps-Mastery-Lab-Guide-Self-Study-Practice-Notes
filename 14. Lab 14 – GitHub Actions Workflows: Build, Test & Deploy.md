# TASK SET 1 – Guided

## Exercise 1 – Create a Basic GitHub Actions Workflow

Create the workflow directory.

```bash
mkdir -p .github/workflows
```

Create a file named **hello.yml**.

```yaml
name: Hello Actions

on:
  push:
    branches:
      - main

jobs:
  greet:
    runs-on: ubuntu-latest

    steps:
      - run: echo "Hello Actions!"
```

Commit and push the workflow to the **main** branch.

Navigate to the **Actions** tab in GitHub and verify that the workflow executes successfully.

> **📌 Note:** GitHub automatically detects workflow files stored under the `.github/workflows` directory.

---

## Exercise 2 – Configure GitHub Secrets

Add the following repository secrets.

1. Navigate to **Repository → Settings**.
2. Select **Secrets and variables**.
3. Choose **Actions**.
4. Create the following secrets:

* `AWS_ACCESS_KEY_ID`
* `AWS_SECRET_ACCESS_KEY`

> **🔒 Security Tip:** Never store AWS credentials directly in workflow files or source code.

---

## Exercise 3 – Create a Continuous Integration (CI) Workflow

Create a workflow that checks out the source code, installs dependencies, and runs automated tests.

```yaml
name: CI Pipeline

on:
  push:

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v4
        with:
          python-version: "3.11"

      - run: pip install -r requirements.txt

      - run: pytest
```

Push the workflow and verify that the pipeline completes successfully.

# TASK SET 2 – Practice

## Exercise 1 – Configure a Matrix Build

Modify the workflow to execute tests on multiple Python versions.

```yaml
strategy:
  matrix:
    python-version:
      - "3.9"
      - "3.10"
      - "3.11"

steps:
  - uses: actions/setup-python@v4
    with:
      python-version: ${{ matrix.python-version }}
```

Verify that separate workflow jobs execute for each Python version.

## Exercise 2 – Upload Workflow Artifacts

Add an artifact upload step.

```yaml
- uses: actions/upload-artifact@v3
  with:
    name: test-results
    path: test-results/
```

After the workflow completes, download the uploaded artifact from the **Actions** page.

> **💡 Tip:** Artifacts are useful for preserving logs, reports, and test results generated during workflow execution.

## Exercise 3 – Cache Python Dependencies

Add dependency caching to improve workflow performance.

```yaml
- uses: actions/cache@v3
  with:
    path: ~/.cache/pip
    key: ${{ runner.os }}-pip-${{ hashFiles('requirements.txt') }}
```

Compare workflow execution times before and after enabling caching.

# TASK SET 3 – Challenge

## Exercise 1 – Create a Complete CI/CD Pipeline

Create a workflow that performs testing, builds a Docker image, pushes it to Amazon ECR, and deploys it to Amazon ECS.

```yaml
on:
  push:
    branches:
      - main

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - run: pytest

      - uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ap-south-1

      - run: |
          aws ecr get-login-password | docker login --username AWS --password-stdin $ECR
          docker build -t $ECR:$GITHUB_SHA .
          docker push $ECR:$GITHUB_SHA
          aws ecs update-service --cluster prod --service my-app --force-new-deployment
```

Verify that the deployment completes successfully after merging changes into the **main** branch.

## Exercise 2 – Integrate SonarCloud

Add a SonarCloud analysis step.

```yaml
- uses: SonarSource/sonarcloud-github-action@master
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
    SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
```

Configure the workflow to fail if the SonarCloud Quality Gate is not passed.

> **📌 Note:** Ensure that the `SONAR_TOKEN` secret has been added to the repository before running the workflow.

## Exercise 3 – Create a Reusable Workflow

Create a reusable workflow named **reusable-build.yml**.

```yaml
on:
  workflow_call:
    inputs:
      image-tag:
        required: true
        type: string
```

Call the reusable workflow from another workflow.

```yaml
jobs:
  build:
    uses: ./.github/workflows/reusable-build.yml
    with:
      image-tag: ${{ github.sha }}
```

> **💡 Tip:** Reusable workflows reduce duplication and promote consistency across multiple repositories and projects.

# 💡 Best Practices

* Store workflow files under `.github/workflows`.
* Protect sensitive credentials using GitHub Secrets.
* Use matrix builds to validate applications across multiple runtime versions.
* Cache dependencies to improve workflow performance.
* Upload test reports and logs as workflow artifacts.
* Trigger deployment workflows only after successful testing.
* Keep reusable workflows modular and version-controlled.
* Apply least-privilege IAM permissions for deployment credentials.

---

# 🔒 Security Considerations

* Never hardcode AWS credentials or access tokens in workflow files.
* Restrict repository access to authorized users.
* Use GitHub Secrets for all sensitive values.
* Rotate access keys and tokens regularly.
* Limit deployment permissions using IAM roles and policies.
* Protect production branches using required reviews and status checks.
