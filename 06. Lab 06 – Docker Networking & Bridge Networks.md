# Lab 06 – Docker Networking & Bridge Networks

# 📖 Overview

Docker networking enables secure communication between containers while providing isolation between applications. In this lab, you will create custom bridge networks, test Docker's built-in DNS resolution, inspect network configurations, implement network isolation, and deploy a three-tier application using Docker Compose.

By completing this lab, you will learn how to:

* ✅ Create custom Docker bridge networks.
* ✅ Connect containers to one or more networks.
* ✅ Resolve containers using Docker's built-in DNS.
* ✅ Inspect Docker network configurations.
* ✅ Enforce network isolation between application tiers.
* ✅ Deploy multi-tier applications using Docker Compose.
* ✅ Restrict network communication using firewall rules.

# TASK SET 1 — Guided

## Exercise 1 — Create a Custom Bridge Network

Create a custom bridge network named **lab-net**.

```bash
docker network create lab-net
```

Deploy an Nginx container on the custom network.

```bash
docker run -d --name nginx --network lab-net nginx
```

Run a temporary Curl container on the same network and access the Nginx container by its hostname.

```bash
docker run -it --rm --network lab-net curlimages/curl curl http://nginx
```

> 💡 Docker automatically provides DNS-based name resolution for containers connected to the same user-defined bridge network.

---

## Exercise 2 — Verify Container Name Resolution (DNS Test)

The previous command already verifies Docker DNS functionality.

The Curl container resolves **nginx** using the container name instead of an IP address.

> **📌 Note:** Container names automatically act as DNS hostnames on user-defined bridge networks.

---

## Exercise 3 — Inspect the Docker Network

Display detailed information about the network and identify the IP addresses assigned to each container.

```bash
docker network inspect lab-net
```

Verify that both containers are connected to **lab-net** and have unique IP addresses.

# TASK SET 2 — Practice

## Exercise 1 — Create Frontend and Backend Networks

Create two isolated Docker networks.

```bash
docker network create frontend-net

docker network create backend-net
```

Deploy an application container connected to the frontend network.

```bash
docker run -d --name app --network frontend-net nginx
```

Connect the application container to the backend network.

```bash
docker network connect backend-net app
```

Deploy a MySQL database on the backend network.

```bash
docker run -d --name db --network backend-net mysql:8 -e MYSQL_ROOT_PASSWORD=pass
```

> 💡 The application container is connected to both networks, allowing it to communicate with frontend clients and the backend database.

---

## Exercise 2 — Verify Network Isolation

Run a temporary Curl container on the frontend network.

Attempt to communicate directly with the database container.

```bash
docker run -it --rm --network frontend-net curlimages/curl curl http://db
```

### Expected Result

```text
Connection failed
```

> ⚠️ The request should fail because the **db** container is not connected to the **frontend-net** network.

---

## Exercise 3 — Run Nginx Using Host Networking

Deploy an Nginx container using the host network.

```bash
docker run -d --name nginx-host --network host nginx
```

Verify that Nginx is accessible from the host.

```bash
curl http://localhost:80
```

> **📌 Note:** Host networking allows the container to use the host machine's network stack directly.

# TASK SET 3 — Challenge

## Exercise 1 — Deploy a Three-Tier Application

Create separate frontend and backend networks.

```bash
docker network create front-net

docker network create back-net
```

Deploy the Redis cache on the backend network.

```bash
docker run -d --name redis --network back-net redis
```

Deploy the Flask backend application.

```bash
docker run -d --name flask --network back-net my-flask-app
```

Connect the Flask application to the frontend network.

```bash
docker network connect front-net flask
```

Deploy the Nginx frontend.

```bash
docker run -d --name nginx -p 80:80 --network front-net nginx
```

The resulting architecture is:

```text
Internet
    │
    ▼
Nginx (Frontend)
        │
        ▼
Flask (Backend)
        │
        ▼
Redis (Cache)
```

---

## Exercise 2 — Create a Docker Compose File

Create a file named **docker-compose.yml**.

```yaml
services:
  nginx:
    image: nginx
    ports:
      - "80:80"
    networks:
      - front

  flask:
    build: .
    networks:
      - front
      - back

  redis:
    image: redis
    networks:
      - back

networks:
  front:

  back:
```

Deploy the application stack.

```bash
docker compose up -d
```

> 💡 Docker Compose automatically creates the required networks and connects the containers based on the configuration.

---

## Exercise 3 — Restrict Network Communication Using `iptables`

Configure a network policy so that the application can communicate with the database only on TCP port **3306**.

```bash
sudo iptables -I DOCKER-USER -s <app-ip> -d <db-ip> ! -p tcp --dport 3306 -j DROP
```

> **📝 Note:** Replace `<app-ip>` and `<db-ip>` with the appropriate container IP addresses.

> 🔒 This rule blocks all traffic except MySQL traffic on port **3306**.

# 💡 Best Practices

* Use user-defined bridge networks instead of the default bridge network.
* Separate application tiers into dedicated networks.
* Connect containers only to the networks they require.
* Use Docker DNS instead of hardcoded IP addresses.
* Deploy multi-container applications using Docker Compose.
* Apply firewall rules to limit unnecessary communication.
* Minimize network exposure by publishing only required ports.

---

# 🔒 Security Considerations

* Avoid exposing backend services directly to the host.
* Restrict database access to trusted application containers only.
* Use Docker bridge networks to isolate workloads.
* Implement firewall policies using `iptables` or cloud security groups.
* Publish only the ports required for client access.
* Regularly inspect Docker networks and container connectivity.
