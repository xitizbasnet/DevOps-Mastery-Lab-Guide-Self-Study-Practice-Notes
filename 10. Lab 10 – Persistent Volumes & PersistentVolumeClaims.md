# Lab 10 – Persistent Volumes & PersistentVolumeClaims

## 📖 Overview

This lab introduces persistent storage in Kubernetes using **Persistent Volumes (PV)** and **Persistent Volume Claims (PVC)**. You will provision persistent storage using the **gp2 StorageClass** on Amazon EKS, mount storage into pods, verify data persistence after pod recreation, deploy stateful applications, expand storage capacity, and implement automated database backups.

By completing this lab, you will learn how to:

* Provision Persistent Volume Claims (PVCs).
* Mount persistent storage into pods.
* Verify data persistence after pod deletion.
* Deploy MySQL with persistent storage.
* Configure shared storage using Amazon EFS.
* Inspect PVC bindings.
* Automate database backups using CronJobs.
* Expand PVC storage capacity.
* Deploy StatefulSets with dynamic volume provisioning.

# TASK SET 1 – Guided

## Exercise 1 – Create a Persistent Volume Claim

Create a file named **pvc.yaml**.

```yaml
apiVersion: v1
kind: PersistentVolumeClaim

metadata:
  name: app-pvc
  namespace: dev

spec:
  accessModes:
    - ReadWriteOnce

  storageClassName: gp2

  resources:
    requests:
      storage: 5Gi
```

Apply the PVC.

```bash
kubectl apply -f pvc.yaml
```

Monitor the claim status.

```bash
kubectl get pvc -n dev -w
```

Observe the status transition:

```text
Pending → Bound
```

> **📌 Note:** The PVC remains in the **Pending** state until a compatible Persistent Volume is dynamically provisioned or matched.

---

## Exercise 2 – Mount the PVC and Write Data

Access the pod that uses the PVC.

```bash
kubectl exec -it app-with-storage -n dev -- sh -c 'echo test > /data/a.txt'
```

This command creates a file named **a.txt** inside the mounted persistent volume.

---

## Exercise 3 – Verify Data Persistence

Delete the pod.

```bash
kubectl delete pod app-with-storage -n dev
```

Recreate the pod.

```bash
kubectl apply -f pod-with-pvc.yaml
```

Verify that the file still exists.

```bash
kubectl exec -it app-with-storage -n dev -- cat /data/a.txt
```

Expected output:

```text
test
```

> **💡 Tip:** Persistent Volumes retain data independently of the pod lifecycle.

# TASK SET 2 – Practice

## Exercise 1 – Deploy MySQL Using a Persistent Volume Claim

Deploy the MySQL application.

```bash
kubectl apply -f mysql-pvc.yaml
```

Create a database.

```bash
kubectl exec -it mysql-pod -n dev -- mysql -uroot -proot123 -e "CREATE DATABASE testdb;"
```

Delete the pod.

```bash
kubectl delete pod mysql-pod -n dev
```

Recreate the pod.

```bash
kubectl apply -f mysql-pod.yaml
```

Verify that the database still exists.

```bash
kubectl exec -it mysql-pod -n dev -- mysql -uroot -proot123 -e "SHOW DATABASES;"
```

> **📌 Note:** The database remains available because the data is stored on the persistent volume rather than inside the container filesystem.

## Exercise 2 – Configure ReadWriteMany Storage Using Amazon EFS

Install the Amazon EFS CSI Driver.

```bash
kubectl apply -k github.com/kubernetes-sigs/aws-efs-csi-driver/deploy/kubernetes/overlays/stable/
```

Create a PVC with the following access mode.

```yaml
accessModes:
  - ReadWriteMany
```

Mount the same PVC into two different pods.

Verify that both pods can read and write to the shared storage simultaneously.

> **💡 Tip:** Amazon EBS supports **ReadWriteOnce**, while Amazon EFS supports **ReadWriteMany**.

# TASK SET 3 – Challenge

## Exercise 1 – Automate Database Backups

Create a CronJob that performs a MySQL backup every hour.

```yaml
apiVersion: batch/v1
kind: CronJob

metadata:
  name: db-backup
  namespace: dev

spec:
  schedule: "0 * * * *"

  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup
            image: mysql:8

            command:
            - /bin/sh
            - -c

            args:
            - mysqldump -h db -uroot -proot123 --all-databases > /backup/dump-$(date +%F).sql

            volumeMounts:
            - name: backup-vol
              mountPath: /backup

          volumes:
          - name: backup-vol
            persistentVolumeClaim:
              claimName: backup-pvc

          restartPolicy: OnFailure
```

Apply the CronJob and verify that scheduled backup jobs execute successfully.

---

## Exercise 2 – Expand Persistent Storage

Increase the PVC size from **5Gi** to **10Gi**.

```bash
kubectl patch pvc app-pvc -n dev -p '{"spec":{"resources":{"requests":{"storage":"10Gi"}}}}'
```

Verify the updated storage size.

```bash
kubectl get pvc app-pvc -n dev
```

> **📌 Note:** The associated StorageClass must have **allowVolumeExpansion** enabled.

---

## Exercise 3 – Deploy a StatefulSet

Create a StatefulSet using **volumeClaimTemplates** so that each replica automatically receives its own Persistent Volume Claim.

Scale the StatefulSet.

```bash
kubectl scale statefulset mysql -n dev --replicas=2
```

Verify the automatically created PVCs.

```bash
kubectl get pvc -n dev
```

Expected output:

```text
mysql-data-mysql-0
mysql-data-mysql-1
```

> **💡 Tip:** Each StatefulSet replica receives a dedicated Persistent Volume Claim, ensuring stable and isolated storage.

# 💡 Best Practices

* Use Persistent Volume Claims instead of static Persistent Volumes whenever possible.
* Select the appropriate StorageClass for your workload requirements.
* Use Amazon EBS for block storage and Amazon EFS for shared file storage.
* Enable automatic volume expansion when supported.
* Store databases on persistent storage rather than container filesystems.
* Regularly back up persistent data.
* Monitor storage utilization and capacity.
* Use StatefulSets for applications requiring stable storage identities.

---

# 🔒 Security Considerations

* Encrypt persistent storage where supported.
* Restrict access to Persistent Volume Claims using Kubernetes RBAC.
* Use dedicated StorageClasses for production workloads.
* Secure database credentials using Kubernetes Secrets instead of hardcoding passwords.
* Regularly validate backup integrity and recovery procedures.
* Limit access to backup storage locations.
